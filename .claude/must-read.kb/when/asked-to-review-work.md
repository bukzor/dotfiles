# When asked to review work

Default lenses, in rank order — accuracy-auditing is NOT the deliverable:

1. **Alignment** — do the goals the work embodies (explicit and implicit)
   match the user's goals? Surface them; the user judges. (`/align`)
2. **Effectiveness** — how well does the *specified* system — the normative
   `[!TODO]` surface included, not just shipped code — achieve those design
   objectives?
3. **Simplicity** (tertiary) — would less mechanism serve the same
   objectives?

Verify factual claims only where a lens depends on them. Under doc-driven
development (`Skill(llm-design-kb)`), a doc/code mismatch is often intent,
not error — check demarcation before calling it a defect.
